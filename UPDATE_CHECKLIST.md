# Monthly update checklist

A light, sustainable rhythm keeps this repository (and my site) visibly alive. The goal is
**one small, real contribution each month** — not a big push before a deadline. Pick *one* line
below; 30–60 minutes is enough.

## The monthly move (pick one)

- [ ] **A Note** — read one AI-hydrology / physical-geography paper and add a 3–5 sentence,
      dated entry to the `Notes` section of the website. What was the idea, and would it work on
      a Taiwan catchment?
- [ ] **A small method demo** — add or extend a notebook cell: a new feature, a new metric, a new
      plot. Even a 10-line improvement counts.
- [ ] **A real-data swap** — point an existing notebook at a Taiwan station (CWB / WRA open data)
      using the `load_taiwan_station()` stub in notebook 01.
- [ ] **A roadmap tick** — convert one "planned" item in the README roadmap into a real cell.

## Before committing

- [ ] Notebook runs top-to-bottom (`Restart & Run All`) with no errors.
- [ ] Any new figure is saved to `assets/` and referenced from the README or site.
- [ ] Commit message is specific (e.g. `add antecedent-wetness feature to rainfall notebook`,
      not `update`).
- [ ] If the site changed, bump "Last updated" in `index.html`.

## Quarterly (every ~3 months)

- [ ] Refresh the README headline metrics if models changed.
- [ ] Add one genuinely new notebook or analysis.
- [ ] Re-read the repo as a hiring committee would: is the strongest work visible in the first
      screen of the README and the site?

---

*Why this matters:* steady, dated activity signals an ongoing scholarly identity. A repo that
gains a commit most months — and a site with fresh notes — reads very differently from one
assembled the week before an application.
